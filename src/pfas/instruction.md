# `src/pfas` Directory Structure & Instructions

This directory contains LAMMPS simulations specifically designed for simulating Per- and Polyfluoroalkyl Substances (PFAS) at various interfaces (like Air-Water).

Because these simulations involve complex molecular topologies (bonds, angles, dihedrals) and custom forcefield parameters, the files can quickly become disorganized. 

**All AI Agents must adhere to the following directory structure:**

## 📂 Directory Structure

### 1. `scripts/` (or root `src/pfas/`)
- Contains the main executable `.lammps` input scripts (e.g., `pfoa_in_water.lammps`).
- *Rule:* Do not clutter the root directory with data files or parameter files.

### 2. `data/`
- Contains all molecular topology files (`.data`) required to build the simulation boxes.
- Examples: `PFOA.data`, `PFOA_stripped.data`.
- *Rule:* If an agent generates a new molecule using Moltemplate, Packmol, or by stripping coefficients, the resulting `.data` file MUST be placed here.

### 3. `ff_params/` (Forcefield Parameters)
- Contains files defining specific forcefield coefficients (e.g., OPLS-AA, GAFF) that are included in the main LAMMPS scripts.
- Examples: `pfoa_coeffs.in`.
- *Rule:* Use `include ff_params/pfoa_coeffs.in` in your `.lammps` scripts instead of hardcoding dozens of coefficients directly into the main script.

### 4. `output/`
- Contains the raw output data generated by running the simulations.
- Organized by simulation name (e.g., `output/pfoa/`).
- *Warning:* These files (like `.lammpstrj`) can be massive. Do not copy them around unnecessarily.

---

## 🤖 AI Workflow Rules for this Folder

1. **Relative Paths**: When editing a `.lammps` script in `src/pfas/`, ensure your `read_data` and `include` commands point to the correct subdirectories (e.g., `read_data data/PFOA_stripped.data`).
2. **Water Models**: If you need the standard 1H2O.data block, reference it from the test folder: `read_data ../test/1H2O.data`.
3. **Execution**: Always run LAMMPS from the root of `src/pfas/` so relative paths resolve correctly. **CRITICAL**: Always use the `lmp_mpi` executable (e.g., `mpirun -np 8 lmp_mpi -in script.lammps`) instead of `lmp`.
4. **Log Failures**: If your simulation crashes because of a parameter combination (e.g., a specific `kspace_style` used with a specific `boundary`), you **MUST** document the failure in `docs/do not work/summery.md` (or create a dedicated file there if it's complex) so future agents don't repeat your mistake.
